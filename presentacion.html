01 · Contexto

El problema de negocio

¿Qué queremos predecir?

La variable objetivo Pago_atiempo indica si un cliente pagará su crédito a tiempo (1) o no (0).

  • Ayuda a las entidades financieras a minimizar el riesgo de cartera.
  • Permite aprobar o rechazar solicitudes automáticamente.
  • Reduce la intervención manual en la calificación de crédito.

Objetivo técnico

  • Construir un pipeline de extremo a extremo: datos → modelo → API.
  • Comparar un modelo heurístico con modelos de ML supervisados.
  • Garantizar reproducibilidad y monitoreo en producción.
  • Métricas principales: Recall y F1-score (penaliza falsos negativos).

Variables derivadas creadas

  • relacion_deuda_ingreso = (otros_préstamos + capital) / salario
  • carga_pago_mensual = cuota / salario
  • ratio_interes_total = (cuota × plazo − capital) / capital
  • grupo_edad_cliente = bins: 18-30, 31-45, 46-60, 61-80
  • cant_creditos_por_sector = Σ créditos financiero + cooperativo + real

Decisiones de diseño

  • Cada transformador implementa sklearn API (fit / transform).
  • KNNImputer con StandardScaler interno para evitar sesgo de escala.
  • n_features_in_ en cada fit() para compatibilidad con sklearn 1.8 check_is_fitted.
  • Fábrica make_pipeline_ml() garantiza instancias frescas y evita estado compartido.

04 · Baseline

Modelo Heurístico

Lógica de decisión

# Riesgo ALTO → predice 0
if (puntaje < -0.4
    or mora > 0.5
    or deuda_ingreso > 0.7
    or carga_pago > 0.8):
    predict(0)

# Perfil estable → predice 1
elif is_empleado and puntaje >= -0.4:
    predict(1)

else: predict(1)

Todos los umbrales son hiperparámetros configurables en __init__.

Evaluación (CV × 10 folds)

MétricaCV MeanTrain
Accuracy ~0.62 ~0.63
F1-score ~0.73 ~0.73
Precision ~0.63 ~0.64
Recall ~0.88~0.88

✅ Recall alto: captura la mayoría de los buenos pagadores.

⚠️ Precisión baja: muchos falsos positivos (aprueba clientes riesgosos).

📌 Sirve como baseline interpretable para comparar contra ML.

07 · Entrenamiento

Proceso de entrenamiento y validación

Validación cruzada

  • KFold(n_splits=10, shuffle=True) sobre el conjunto de entrenamiento.
  • Métricas reportadas: accuracy, precision, recall, F1.
  • Curvas de aprendizaje con ShuffleSplit(50 splits) para análisis de bias/varianza.
  • Análisis de escalabilidad: fit_time y score_time por tamaño de dataset.

Pipeline de entrenamiento

Pipeline([
  ("features", make_pipeline_ml()),
  ("model",    RandomForestClassifier(...))
])

Artefactos generados

  • artifacts/best_model.joblib — modelo serializado listo para producción.
  • artifacts/model_summary.csv — tabla comparativa de todos los candidatos.
  • artifacts/model_comparison.png — gráfico de barras: F1, CV std, fit time.

Observación de curvas de aprendizaje

La brecha train/CV se cierra con más datos → el modelo generaliza bien y no hay overfitting severo.

08 · Despliegue

API de inferencia con FastAPI

Arquitectura

  • FastAPI expone un endpoint POST /predict para predicción por lotes.
  • Pydantic valida el esquema de entrada y salida.
  • Retorna predictions: List[int] y probabilities: List[float].
  • El modelo se carga desde artifacts/best_model.joblib al iniciar.

Dockerfile

FROM python:3.11-slim
EXPOSE 8000
CMD ["uvicorn", "model_deploy:app",
     "--host", "0.0.0.0",
     "--port", "8000"]

Payload de ejemplo

// POST /predict
{
  "records": [
    {
      "puntaje_datacredito": -0.2,
      "saldo_mora":          0.0,
      "salario_cliente":     3500000,
      "capital_prestado":    10000000,
      ...
    }
  ]
}

// Respuesta
{
  "n_records":     1,
  "predictions":  [1],
  "probabilities":[0.82]
}

10 · Monitoreo

Monitoreo continuo con PSI

Population Stability Index (PSI)

Mide el cambio en la distribución de cada feature entre el baseline (entrenamiento) y la data de producción.

PSIInterpretación
< 0.10Sin drift — estable
0.10 – 0.20Monitorear — cambio menor
> 0.20🚨 Drift significativo — reentrenar

Soporta variables numéricas (bins por cuantiles) y categóricas (frecuencias).

Ciclo de monitoreo

  • Se toman muestras periódicas de datos en producción.
  • Se calcula PSI por feature y se marca drift_flag=True si PSI ≥ 0.20.
  • Se registran métricas de rendimiento en tiempo real (con etiquetas disponibles).
  • Logs acumulados en CSVs append-mode.

Artefactos de monitoreo

  • monitoring/drift_metrics.csv — PSI por feature y run
  • monitoring/performance_metrics.csv — métricas en el tiempo
  • monitoring/prediction_log.csv — log de predicciones
  • monitoring/latest_run_summary.json

  "monitoring_sample_size":          500,

Beneficios

  • Un único lugar para cambiar rutas, umbrales y parámetros.
  • El script setup.bat lee project_code para crear el virtualenv con el nombre correcto.
  • Elimina valores hardcoded dispersos en 4 módulos Python.
  • Facilita la promoción entre entornos (dev → staging → prod).

Uso en Python

import json
from pathlib import Path

    return json.loads(
        (Path(__file__).parent / "config.json")
        .read_text()
    )
